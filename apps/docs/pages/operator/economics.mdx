# Economics & Gas Analysis

SWR stores every registration as a permanent on-chain entry. Each entry struct is optimized to fit in a single EVM storage slot (17-26 bytes depending on registry type), containing only the fields needed for on-chain lookups. Rich provenance data (chain IDs, reporters, message IDs) is emitted in events for indexer consumption. On Ethereum L1, even this minimal storage would cost dollars per entry. On Base, the economics are orders of magnitude cheaper — and Base continues to scale toward higher gas limits and lower fees.

## L2 Storage Economics

The fundamental design decision: store every entry on-chain as a single-slot struct rather than relying on off-chain aggregation. Each entry occupies exactly one storage slot (WalletEntry: 26 bytes, TransactionEntry: 18 bytes, ContractEntry: 17 bytes). L2 gas economics make per-entry storage practical.

The per-transaction gas limit constrains how many entries fit in a single batch transaction. At ~26,000 gas per entry (measured via forge gas tests), the practical maximum is **~950 entries per transaction**. Larger datasets must be split across multiple transactions. ### Per-Entry Gas Cost

Each entry in a batch writes one storage struct, emits one event, and performs skip checks. From forge gas measurements against the current contracts:

| Operation                    | Gas         | What it does                                              |
| ---------------------------- | ----------- | --------------------------------------------------------- |
| Cold SSTORE (zero → nonzero) | ~22,100     | Write one entry struct (1 slot per entry)                 |
| Event emission               | ~2,000      | WalletRegistered / TransactionRegistered / etc.           |
| Misc (keccak, SLOAD, loop)   | ~2,100      | Storage key hash, duplicate check, array access           |
| **Per-entry total**          | **~26,200** | Measured across wallet, transaction, and contract batches |

Batch overhead adds ~65,000 gas for the summary event, batch metadata storage, and function dispatch (one-time per batch).

## Why Base?

SWR uses Base as the hub chain for settlement and operator submissions. The core reason: **gas economics make single-slot on-chain storage feasible.**

SWR's design stores every entry fully on-chain — one EVM storage slot per entry (17-26 bytes). On Ethereum L1, this would cost dollars per entry. On Base, L2 gas economics make per-entry storage practical at scale — from individual self-attestations to operator batch submissions. Base is also actively scaling toward higher throughput and lower fees, which further improves the economics over time.

| Factor               | Base                                                                            |
| -------------------- | ------------------------------------------------------------------------------- |
| TVL                  | Multi-billion (check [DeFiLlama](https://defillama.com/chain/Base) for current) |
| Monthly transactions | Tens of millions (check [BaseScan](https://basescan.org/chart) for current)     |
| Block time           | ~2 seconds                                                                      |
| Transaction cost     | Sub-cent (varies with L1 blob fees)                                             |
| Stack                | OP Stack (Optimism Superchain ecosystem)                                        |

Single-slot storage also means the registry is **fully queryable on-chain**. Any smart contract can call `isRegistered(address)` and get a boolean back in the same transaction — enabling downstream contracts (off-ramps, bridges, DeFi protocols) to gate operations based on fraud status. Alternative approaches like merkle root storage or SSTORE2/SSTORE3 blob packing would make data available to indexers and off-chain consumers, but contracts couldn't ask "is this wallet stolen?" without off-chain proof submission. Per-entry storage on an L2 is the only design that gives you both on-chain composability and practical economics.

## Protocol Fee Structure

The `FeeManager` contract manages registration fees denominated in USD and converted to ETH via Chainlink oracle:

| Fee Type                | USD   | Who Pays                                           |
| ----------------------- | ----- | -------------------------------------------------- |
| Individual registration | $5.00 | Any user registering a wallet or transaction batch |
| Operator batch          | Free  | DAO-approved operators pay only gas                |

Individual fees serve as an anti-spam mechanism — they create a cost barrier that makes mass false registrations expensive. Operators are DAO-approved entities whose trust is established through the approval process itself, so no additional fee is required.

### USD-to-ETH Conversion

```solidity
function currentFeeWei() public view returns (uint256) {
    return (baseFeeUsdCents * 1e18) / ethPriceUsdCents;
}
```

At ETH = $3,000: `(500 * 1e18) / 300000 = 0.00167 ETH` (~$5.00)

### Price Oracle

The `FeeManager` uses a Chainlink ETH/USD price feed with a manual fallback:

| Parameter                  | Default           | Purpose                                        |
| -------------------------- | ----------------- | ---------------------------------------------- |
| `stalePriceThreshold`      | 14,400s (4 hours) | Max age before falling back to stored price    |
| `fallbackSyncInterval`     | 86,400s (1 day)   | How often to auto-sync fallback from Chainlink |
| `fallbackEthPriceUsdCents` | 300,000 ($3,000)  | Manual price used when oracle is unavailable   |

If Chainlink is stale or unreachable, the contract uses the fallback price. The fallback is opportunistically synced from Chainlink during normal operations, so it stays reasonably fresh without extra transactions.

## Operator Incentives

Why would an exchange or security firm contribute fraud data to a public registry?

**Shared intelligence.** When one exchange reports a stolen wallet, every other exchange, wallet, and off-ramp benefits. The cost of NOT sharing — processing withdrawals to known stolen wallets — far exceeds the negligible gas cost of submitting data on an L2.

**Regulatory signaling.** Demonstrating active fraud prevention to regulators. Submitting to a public, verifiable registry is stronger evidence than internal-only systems.

**Community trust.** Users prefer exchanges that actively contribute to ecosystem security. Operator status is public and verifiable on-chain.

**Zero protocol fees.** Operators pay only L2 gas — no protocol fees. Submit your fraud data for free, get everyone else's data in return.

## Future: Operator-as-Paymaster

The long-term vision is EIP-4337 Account Abstraction integration where operators act as Paymasters for individual registrations:

    EX->>U: Present EIP-712 signature request
    U->>EX: Sign acknowledgement
    Note over EX: Grace period (~2 min)
    EX->>U: Present second signature request
    U->>EX: Sign registration
    EX->>PM: Bundle with Paymaster sponsorship
    PM->>SWR: Submit registration (operator pays gas)
    SWR-->>U: Wallet registered. Zero cost to user.

In this model:

- Users sign EIP-712 messages on the exchange's site
- The exchange's Paymaster contract pays gas
- Registration is **free for the user**
- The exchange gets the fraud data they want, the user gets free registration

This creates a flywheel: more operators -> more free registrations -> more data -> more operators.
